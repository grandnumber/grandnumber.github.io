TDD or Test Driven Development because acronyms make everything more serious...

24th January 2015

"TDD" as a term is something I have heard bandied about for a long time. Before I started learning to code my general reaction to hearing people talk of Test Driven Development in awe was "huh? so you didn't bother testing things before? "... Needless to say I have a greater appreciation for the power of TDD and why it isn't simply putting tests in where there were none before ;-)

To be precise, TDD is: "a software development process that relies on the repetition of a very short development cycle: first the developer writes an (initially failing) automated test case that defines a desired improvement or new function, then produces the minimum amount of code to pass that test, and finally refactors the new code to acceptable standards. " Which in simple terms means a 5 step process:

1. Write Test:

Write a test for the smallest part of code that you need to write.

2. Run & Fail:

Run the test - it will fail.. given there is no code ;-)

3. Write Code:

Now, write code to pass the test. But focus on the minimum required to pass your test - no prizes for sophistication or extra functionality - just focus on passing the test!

4. Run & Pass:

Now, run the test again - and make sure it passes! Don't write any new code or functionality until this first test passes. Once it passes - move on..

5. Refactor:

And refactor if necessary/appropriate at this time... then you get to

6. Repeat:

For every portion of your code!!!
