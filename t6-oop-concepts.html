Ruby variable scoping sometimes 'computah says NO'...

10th January 2014

In Ruby variable scoping allows for variables to be accessible (or not!) in specific situations. This helps protect data by making it accessible only at certain levels in your program. There are 5 variables with attendant scope options:

1. CONSTANTS

No I'm not shouting... but constants do. They are indicated by an ALLCAPS naming convention CONSTANT Constant variables intended to be set once (and only updated at set intervals rather than be updated throughout the program). The intent to update them rarely is reinforced by Ruby which issues a warning message anytime you try to change a constant variable. They are accessible from anywhere within the place they were set. If they are set outside an object, they are accessible globally throughout a program. If they are set within a class or method, they are accessible only within that class or method. :



  SOURCE_DATA = {"important"=>"stuff", "more_important=>stuff"}

2. Global Variables

Let's get this out of the way - global variables are dangerous! Global variables are accessible from anywhere within the program, regardless of where they are declared. Unlike constants, global variables can be changed easily without a warning message - making debugging difficult. They are denoted with a dollar sign (“$”) in front of the variable name (which makes them harder to 'accidentally' overwrite than in Javascript at least!).

3. Class Variables

Class variables are shared among all instances of a particular class (Ruby classes were discussed in the last post here ). Given that Class Variables sharing approach to scope if an object changes the class variable, it will be changed for all instances of the class variable later on. Class variables are denoted with @@ before the variable name (e.g., @@car_speed) and are initialized when the class is created.

4. Instance Variables

Instance variables. Denoted with @ before the variable name (e.g., @car_color), instance variables are local to specific instances of an object. They let you use the same variable across all the methods within your class. For example you might have the object of a class called Cars and set an instance variable called @car_color which would allow you to set the color of the car in your specific car class and then reference it later methods within that same class.

5. Local Variables

Our lucky last, local variables, are the most basic type of variable. They are named without any prepend (e.g., car). Local variables are only accessible immediately where they are declared. For example, a local variable declared within a method is only accessible within that method.
